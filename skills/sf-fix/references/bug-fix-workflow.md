# Bug Fix Workflow

Use this reference after the top-level `sf-fix` activation contract has selected a proof path and classified the bug.

## Intake Detail

Collect only what is needed: observed behavior, expected behavior, where it happens, repro steps, and error message. Infer a one-line bug story when safe, but confirm briefly if the inference could change behavior or risk.

Prefer decision-forcing questions:

- Should this case block the action, hide it, or show an explicit error?
- Does it affect all signed-in users, admins only, or only the owner?
- On partial failure, should the system retry, rollback, or leave a visible error state?

## BUG-ID Procedure

When arguments include `BUG-YYYY-MM-DD-NNN` or a title matching a bug file, open `bugs/BUG-ID.md` right before intake. Optional `BUGS.md` is secondary discovery or triage context only.

If multiple bugs match, prefer the open one with the most recent `last-tested` date and ask when ambiguous.

If `BUGS.md` has an entry but `bugs/BUG-ID.md` is missing, keep the index row, report `needs-info`, and ask for recovery context or route to `/sf-test --retest BUG-ID`.

When creating a bug:

1. List `bugs/BUG-YYYY-MM-DD-*.md`.
2. Re-read optional `BUGS.md` if present.
3. Choose the next UTC-date suffix greater than every suffix found.
4. Check for file collision immediately before writing; if collision exists, re-read once, increment, and retry.
5. If collision repeats, stop instead of overwriting.
6. Create from `templates/artifacts/bug_record.md`.
7. Refresh optional `BUGS.md` only when that triage view exists or is generated by workflow.

## Bug File Content

Initial or refreshed bug files must include summary/title/status/severity/next step, reproduction, expected behavior, observed behavior, redacted evidence pointers, diagnosis notes, fix attempts, retest history, and redaction status.

When investigation starts, move status to `in-diagnosis` and append a Diagnosis Notes row. Every concrete code attempt appends a Fix Attempts row with timestamp UTC, files changed, hypothesis, proof path, validation command or reason skipped, result, Sentry pointer status when relevant, and next retest command.

If no passing retest exists, status cannot be `closed`.

## Technical Triage

Read only the 3-5 most relevant files and classify the bug as `direct` or `spec-first`.

Before choosing `direct`, verify:

- reproduction or named proof gap
- root cause hypothesis for repeated attempts
- user story fit
- product coherence with adjacent flows, copy, permissions, and conventions
- documentation coherence for visible behavior
- fresh external docs support when external behavior matters
- security posture, including auth/authz, input validation, and data exposure
- local-enough blast radius
- decision quality and excellence

## External Gates

Use documentation freshness when current external behavior matters. Use project development mode before deciding retest surface.

For auth, protected routes, OAuth redirects, Clerk/Supabase session state, callback handling, or code that works statically but fails in browser, use `sf-auth-debug` as diagnostic evidence while keeping `sf-fix` as execution owner.

For non-auth browser reproduction, prefer `sf-browser` evidence before patching when observable runtime behavior matters.

For crashes, error boundaries, 5xx, visible Sentry/support event IDs, production exceptions, or deployed-runtime failures, load Sentry observability. Never paste raw payloads, breadcrumbs, replay contents, headers, cookies, tokens, private URLs, or PII into bug files or reports.

## Direct Fix Execution

When classified `direct`:

- implement the bounded professional repair
- preserve the user story outcome and product coherence
- update or flag docs describing fixed behavior
- preserve security-by-default
- mark task in progress only when a clear tracker row exists
- re-read trackers immediately before edits
- run relevant checks and at least one user-story sanity check
- route preview-push retests through `sf-ship -> sf-prod -> sf-test --preview --retest BUG-ID`
- append a Fix Attempts row
- keep status `fix-attempted` until retest evidence exists
- allow `fixed-pending-verify` only after passing retest
- refuse `closed` without retest evidence
- run `sf-verify` logic after retest before closure

## Spec-First Or Diagnostic-Only

If `spec-first`, do not code. Route to `/sf-spec [bug title]`, `/sf-ready [bug title or spec path]`, then `/sf-start [bug title]`.

If `diagnostic only`, do not code. Route to `sf-auth-debug`, `sf-browser`, or report suspected root cause and a concrete next command.
